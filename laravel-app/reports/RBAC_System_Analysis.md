# Report: Role-Based Access Control and Team Leader View Restrictions in ArchEng Pro

## 1. Overview and Architecture

The application implements a robust, multi-tiered Role-Based Access Control (RBAC) system. The core requirement is that "Team Leaders" (e.g., a manager for the MEP or Architecture department) should only visualize data, metrics, and profiles relevant to their specific subordinates, while "Master Admins" retain an unrestricted, global view of the entire organization.

This system is built around three primary database models and their relationships:
*   **Users (`users` table):** Represents the login credentials, containing their `role` (e.g., `admin`, `team_leader`) and a newly added `department` column to identify which domain a team leader belongs to.
*   **User Profiles (`user_profiles` table):** Represents the tracked employees and candidates. They are assigned to a `department`.
*   **Monitor Assignments (`monitor_assignments` table):** A dynamic pivot table that explicitly links a specific `team_leader` (from `users`) to specific `monitored_user_name` strings (from `user_profiles` or `activity_logs`).

## 2. Dynamic Authorization Logic

The restriction engine is centralized in a helper method called `getAuthorizedUsernames($dept)` within the `DashboardController`. This ensures consistent application of rules across all views.

### How `getAuthorizedUsernames` Works

When a Team Leader accesses the system, the application performs a layered authorization check:

1.  **Direct Assignments Priority:** It first queries the `monitor_assignments` table. If the Master Admin has specifically assigned individual candidates to this Team Leader, the application *only* returns those specific usernames. This allows a leader to monitor a cross-functional team (e.g., 2 MEP users and 1 Architecture user).
2.  **Department Fallback:** If there are no specific assignments (the pivot table is empty for this leader), the system falls back to the Team Leader's profile. If the Team Leader is assigned to "MEP", the system automatically queries the `UserProfiles` table and returns *all* usernames assigned to the "MEP" department.
3.  **Total Restriction:** If the Team Leader has no direct assignments and is not assigned to a department, they are securely restricted and see zero data, ensuring no accidental data leaks.
4.  **Admin Pass-Through:** If the logged-in user is a Master Admin (`role === 'admin'`), the method dynamically returns `null`, signaling to the query builders to apply zero restrictions.

## 3. Query Execution and Data Filtering

The returned list of `$authorizedUsernames` acts as a mandatory filter applied at the uppermost level of almost all database queries in the application.

### Dashboard & Analytics (`DashboardController`)
Every metric on the dashboard is intercepted before execution:
*   **Live Online Users:** The system checks standard activity logs, but appends a `whereIn('user_name', $authorizedUsernames)` clause. The Team Leader only sees a live count of their own team members.
*   **Time Worked & Software Usage:** When calculating total hours or software market share, the identical SQL `IN()` clause filters out logs generated by employees outside the leader's purview.
*   **Leaderboard:** When calculating the top 5 longest-working users, the query groups by user but drops any user not in the `$authorizedUsernames` array before applying the `limit(5)`.

### View Restrictions

*   **Locked Dropdowns:** In the UI (such as the License Audit or Dashboard pages), the application detects the user's role. If the user is a `team_leader`, the department selection dropdown is forcefully overridden. The dropdown displays only the leader's department (or "My Assigned Team") and the backend ignores any forged request attempting to query a different department.
*   **Read-Only Operations:** In the User Management view, `team_leaders` can see the list of users they manage, but Blade directives (e.g., `@if($user->role === 'admin')`) hide administrative actions such as "Assign New License", "Edit Profile", or "Delete User".

## 4. Endpoint Security (`ProfileController`)

The deepest level of restriction occurs when a Team Leader attempts to view an individual's detailed profile.

If a Team Leader navigates to `/profile/{userName}`:
1.  The controller retrieves the `$authorizedUsernames` array.
2.  It strictly checks: `if (!in_array($userName, $authorizedUsernames))`
3.  If the requested username is not in their allowed list, the application instantly throws an HTTP 403 Forbidden Exception (`abort(403, 'Unauthorized. You do not have permission to monitor this user.')`).

This security logic prevents URL manipulation (Direct Object Reference vulnerabilities), ensuring a Team Leader cannot simply guess the username of a candidate in another department and view their metrics.

## 5. Summary

The application successfully achieves strict data silos for team leads while maintaining global access for administrators. By blending **Department-based filtering** with an explicit **Direct Assignment** override, the architecture provides both out-of-the-box convenience (assign a leader to MEP, they see MEP) and highly granular control (cherry-picking specific users for a leader to monitor). All database interactions are gated by this logic.
